# Day 2 知识点清单与详解

## Day 2 学习目标

Day 2 的目标不是进入 Netty 通信主链路，而是把 `rpc-framework-common` 这个底层支撑模块完整复刻并真正理解。这个模块虽然不直接处理一次 RPC 调用，但它决定了整个框架的扩展能力、错误表达方式、工具复用方式和对象管理方式。

今天结束时，你应该达到以下结果：

1. 能独立说明 `rpc-framework-common` 的职责边界。
2. 能解释为什么框架需要枚举、异常、工具类、SPI、单例工厂和线程池工具。
3. 能看懂并复述 `ExtensionLoader` 的完整工作流程。
4. 能说明 `SingletonFactory` 为什么存在，以及它和 SPI 的关系。
5. 能把 Day 2 学到的内容作为 Day 3 主链路复刻的基础讲清楚。

---

## 一、Day 2 复刻范围总览

Day 2 主要复刻 `rpc-framework-common` 模块中的以下部分：

1. `enums`
2. `exception`
3. `utils`
4. `extension`
5. `factory`
6. `utils.concurrent.threadpool`

这些部分虽然不直接发请求、收请求，但它们是整个框架的地基。

你可以把这个模块理解成三层：

1. 表达层  
   用枚举和异常统一表达配置、状态和错误。

2. 支撑层  
   用工具类、线程池工具、运行时工具统一处理通用逻辑。

3. 扩展层  
   用 `SPI + ExtensionLoader + SingletonFactory` 实现框架的可插拔能力。

---

## 二、为什么 Day 2 先做 common 模块

很多人复刻 RPC 项目时，一上来就想写 Netty、写注册中心、写动态代理，但这会有一个明显问题：主链路代码会依赖很多“底层公共能力”，如果这些东西没先搭起来，后面的类会到处报红、逻辑会很散、结构会失去控制。

在这个项目里，`rpc-framework-simple` 中的大量核心类都依赖 `rpc-framework-common`：

1. 依赖错误码和错误信息枚举。
2. 依赖扩展加载器动态选择实现。
3. 依赖单例工厂创建共享实例。
4. 依赖配置读取工具拿到 ZooKeeper 地址等配置。
5. 依赖线程池工具创建统一线程池。

所以 Day 2 的意义是：  
**先把底座搭稳，再进入主调用链。**

### 背诵版

Day 2 先复刻 `rpc-framework-common`，是因为它承载了 RPC 框架的底层公共能力，包括枚举、异常、工具类、SPI 加载器、单例工厂和线程池工具。`rpc-framework-simple` 中的核心类会大量依赖这些基础设施，所以必须先把 common 模块搭好，后面的主链路复刻才会稳定。

## 三、枚举类知识点

Day 2 第一块通常从枚举开始，因为它们实现简单，但对全局理解很重要。

### 1. 枚举在框架中的作用

在业务代码里，很多人习惯直接写字符串，比如：

- `"hessian"`
- `"gzip"`
- `"zk"`
- `"netty"`

但在框架代码里这样写会带来几个问题：

1. 容易拼错。
2. 难以统一管理。
3. 扩展时不方便检索和维护。
4. 很难表达“这是框架中的固定选项”。

所以框架会用枚举统一表示固定概念。

### 2. 这个项目里常见枚举的分类

#### 配置与扩展项类

- `SerializationTypeEnum`
- `CompressTypeEnum`
- `RpcRequestTransportEnum`
- `ServiceRegistryEnum`
- `ServiceDiscoveryEnum`
- `LoadBalanceEnum`

这些枚举用于表达“某个能力使用哪一种实现方式”。

#### 状态与错误表达类

- `RpcResponseCodeEnum`
- `RpcErrorMessageEnum`

这些枚举用于表达“框架运行结果和异常语义”。

#### 配置键名类

- `RpcConfigEnum`

它用于统一配置文件中的 key，避免把配置项名字写死在多个地方。

### 3. 为什么框架里枚举比常量字符串更合适

枚举比常量字符串更适合框架，原因有三点：

1. 语义明确  
   一眼就能看出这是一个固定集合，而不是任意字符串。

2. 扩展清晰  
   新增序列化方式时，只需要在对应枚举和映射逻辑里统一扩展。

3. 更利于排错  
   出问题时能直接定位到枚举分支，而不是在项目里搜大量字符串。

### 4. 重点枚举解释

#### `RpcResponseCodeEnum`

它表示一次 RPC 调用结果的状态码，比如成功、失败。

它的作用：

1. 让客户端可以机器化判断调用是否成功。
2. 让响应不仅有文字信息，还有统一状态码。
3. 让业务层和框架层在错误表达上有统一标准。

#### `RpcErrorMessageEnum`

它表示框架中的典型错误语义，比如：

- 服务未找到
- 请求与响应不匹配
- 服务调用失败

它的作用：

1. 统一错误文案。
2. 让异常包装时更规范。
3. 让后续日志和排错更清晰。

#### `SerializationTypeEnum`

它表示项目支持的序列化方式，比如：

- Kryo
- Protostuff
- Hessian

它的作用：

1. 标识当前消息用了哪种序列化。
2. 在协议层把字节码和具体实现关联起来。
3. 和 SPI 扩展机制配合，根据编码获取实现类。

#### `CompressTypeEnum`

它表示压缩方式，目前项目中主要是 `gzip`。

它的作用：

1. 在协议层告诉对端数据采用了什么压缩方式。
2. 让压缩能力具备可扩展性。

#### `LoadBalanceEnum`

它表示负载均衡策略。

它的作用：

1. 让客户端服务发现后可以按策略选地址。
2. 让随机、轮询、一致性哈希等算法具备可切换性。

### 背诵版

枚举在框架中的作用，是统一表达固定配置项、固定状态和固定错误语义。它比硬编码字符串更安全、更清晰、更易扩展。这个项目里的枚举主要分为扩展项枚举、状态错误枚举和配置项枚举，后续 SPI、注册发现、协议处理和异常包装都会依赖这些统一表达。

## 四、异常类知识点

Day 2 的第二块是异常体系。这个项目的异常类不多，但意义很重要。

### 1. 为什么框架要定义自己的异常

如果一个框架在运行过程中只抛原生异常，比如：

- `NullPointerException`
- `IOException`
- `ClassNotFoundException`
- `IllegalArgumentException`

那调用方很难判断“这到底是框架哪一层出了问题”。  
所以框架通常会包装成自己的异常体系。

### 2. `RpcException` 的作用

`RpcException` 是这个框架里最核心的业务异常包装之一。

它主要用来表示：

1. 服务调用失败
2. 服务未找到
3. 请求与响应不匹配
4. RPC 主流程中的其他逻辑错误

它的意义是：

1. 把 RPC 框架自己的错误语义和 Java 底层异常区分开。
2. 便于上层统一捕获和打印。
3. 让错误信息可以和 `RpcErrorMessageEnum` 对齐。

### 3. `SerializeException` 的作用

序列化和反序列化是 RPC 框架非常独立的一层。

如果这一层失败，比如：

- 对象无法序列化
- 字节流无法反序列化
- 序列化框架内部报错

通常不适合直接抛普通 `RuntimeException`，因为这类错误属于“协议和编码层”的异常。  
所以项目会用 `SerializeException` 专门表达这类问题。

### 4. 为什么异常要和枚举配合

这个项目里，异常不只是“抛出去”，而是通常会结合：

- `RpcErrorMessageEnum`
- `RpcResponseCodeEnum`

这样做的好处是：

1. 异常信息统一。
2. 响应码和日志信息统一。
3. 更适合框架做标准化错误处理。

### 背诵版

框架定义自己的异常，是为了把框架层错误和底层 Java 异常区分开。`RpcException` 用于表达 RPC 调用流程中的错误，比如服务未找到、请求响应不匹配和调用失败；`SerializeException` 用于表达序列化和反序列化阶段的错误。异常和枚举配合使用，可以统一错误语义和状态表达。

## 五、工具类知识点

Day 2 的第三块是工具类。它们看起来简单，但实际是框架中最常见的“胶水层代码”。

### 1. 为什么要有工具类

框架中会反复出现一些小逻辑：

1. 字符串判空
2. 集合判空
3. 获取 CPU 核数
4. 读取配置文件

如果这些逻辑分散在几十个类里，会出现：

1. 重复代码很多
2. 风格不统一
3. 改动成本高

所以工具类的意义是：
**把重复的小逻辑统一封装。**

### 2. `StringUtil`

它主要处理字符串相关的通用判断，比如是否为空、是否为空白。

它的作用：

1. 给 `ExtensionLoader` 等类做参数校验。
2. 避免到处重复写字符串判空逻辑。

### 3. `CollectionUtil`

它主要处理集合判空。

它的作用：

1. 给注册发现和负载均衡逻辑做前置判断。
2. 提高代码可读性。

### 4. `RuntimeUtil`

它通常用于获取运行时信息，比如 CPU 核心数。

它在这个项目里的意义主要是：

1. 给线程池大小设置提供基础依据。
2. 避免直接在多个类里写 `Runtime.getRuntime()`。

### 5. `PropertiesFileUtil`

这是项目里最重要的工具类之一，因为它关系到配置读取。

它的主要作用：

1. 从 `rpc.properties` 中读取配置。
2. 给 ZooKeeper 地址等运行配置提供来源。

它通常会被：

- `CuratorUtils`
- 其他需要读取配置的基础设施类

调用。

### 背诵版

工具类的作用是封装项目中反复出现的小逻辑，减少重复代码并提高一致性。`StringUtil` 负责字符串判空，`CollectionUtil` 负责集合判空，`RuntimeUtil` 负责运行时信息获取，`PropertiesFileUtil` 负责配置文件读取。它们共同构成框架的通用支撑层。

## 六、SPI 机制知识点

这部分是 Day 2 最核心的内容，必须重点理解和背诵。

### 1. 什么是 SPI

SPI，Service Provider Interface，本质上是一种：

**接口先定义好，具体实现可插拔替换的扩展机制。**

在这个项目里，很多能力不是写死的，而是做成扩展点：

1. 序列化方式可以换
2. 压缩方式可以换
3. 注册中心实现可以换
4. 服务发现实现可以换
5. 负载均衡算法可以换
6. 请求传输方式可以换

所以框架只依赖接口，不依赖唯一实现。

### 2. 这个项目为什么必须有 SPI

RPC 框架是一个天然需要扩展的系统，因为：

1. 不同项目可能偏好不同序列化方案
2. 不同环境可能使用不同注册中心
3. 不同业务可能使用不同负载均衡策略
4. 同一个框架未来也可能继续增加实现

如果没有 SPI，每新增一种实现都要硬改代码，框架就会变得僵硬。

### 3. `@SPI` 注解的意义

`@SPI` 是扩展接口的标记。

它告诉框架：

1. 这个接口允许被扩展
2. 可以通过 `ExtensionLoader` 加载它的实现

只有带 `@SPI` 的接口，才被当成真正的扩展点。  
这样做可以避免普通接口被误当作插件接口。

### 4. SPI 配置文件的组织方式

这个项目里，扩展配置不是写在代码里，而是写在：

```text
META-INF/extensions/接口全限定名
```

例如：

```text
META-INF/extensions/github.javaguide.serialize.Serializer
```

文件内容一般是：

```text
hessian=github.javaguide.serialize.hessian.HessianSerializer
protostuff=github.javaguide.serialize.protostuff.ProtostuffSerializer
kyro=github.javaguide.serialize.kyro.KryoSerializer
```

这表示：

- 扩展名 `hessian` 对应某个实现类
- 扩展名 `protostuff` 对应某个实现类
- 扩展名 `kyro` 对应某个实现类

### 5. JDK SPI 和这个项目 SPI 的区别

JDK 原生 SPI 更偏“加载所有实现，再自己遍历选择”。  
这个项目的 SPI 更偏 Dubbo 风格，特点是：

1. 可以按名字拿具体实现
2. 不需要先遍历所有实现
3. 可以缓存实例
4. 更适合策略切换型框架

例如：

```java
ExtensionLoader.getExtensionLoader(Serializer.class).getExtension("hessian")
```

这种按名字加载非常适合 RPC 这种高扩展场景。

### 背诵版

SPI 是一种可插拔扩展机制，框架先定义扩展接口，再通过配置文件把扩展名映射到具体实现类，运行时按名称动态加载。这个项目中，序列化、压缩、注册中心、服务发现、负载均衡和传输方式都通过 SPI 实现可扩展。相比 JDK 原生 SPI，这个项目的 SPI 更接近 Dubbo 风格，支持按名称精确获取扩展实现，并配合缓存和单例工厂提高效率。

## 七、`Holder` 知识点

### 1. `Holder` 是什么

`Holder` 是一个非常轻量的小容器，通常只包一层对象引用。  
虽然它简单，但在框架里很有用。

### 2. 它为什么存在

在 `ExtensionLoader` 和 `SingletonFactory` 这种场景里，经常需要：

1. 先查缓存
2. 没有再创建
3. 创建后放进缓存
4. 多线程下保证可见性和一致性

`Holder` 的作用就是给这种“延迟初始化 + 缓存共享”的场景提供一个对象包装器。

### 3. 为什么不直接存对象

直接存对象也能做，但 `Holder` 的好处是：

1. 逻辑更清晰
2. 适合做双重检查和同步控制
3. 让缓存管理更统一

### 背诵版

`Holder` 是一个轻量级对象包装器，主要用于保存共享实例，配合缓存和同步逻辑实现安全的懒加载。它常用于扩展加载器和单例工厂中管理对象引用。

## 八、`ExtensionLoader` 详解

这是 Day 2 最重要的类。

### 1. 它解决的核心问题

`ExtensionLoader` 的核心作用是：

**根据接口类型和扩展名，动态获取对应实现对象。**

例如：

1. 给我 `Serializer` 接口
2. 给我扩展名 `hessian`
3. 返回 `HessianSerializer` 的实例

### 2. 它的完整工作流程

你必须把它记成 6 步。

#### 第一步：拿到接口对应的加载器

调用：

```java
ExtensionLoader.getExtensionLoader(Serializer.class)
```

这一步会做接口级别的校验。

#### 第二步：校验接口是否合法

通常要检查：

1. 类型不能为空
2. 必须是接口
3. 必须被 `@SPI` 标记

如果不满足，就不允许作为扩展点使用。

#### 第三步：读取扩展配置文件

它会去 `META-INF/extensions/` 下找对应接口全限定名的文件。

例如：

```text
META-INF/extensions/github.javaguide.serialize.Serializer
```

#### 第四步：解析配置

把文件里的：

```text
hessian=github.javaguide.serialize.hessian.HessianSerializer
```

解析成：

```text
扩展名 -> 实现类
```

并缓存起来。

#### 第五步：按扩展名获取类

调用：

```java
getExtension("hessian")
```

时，会：

1. 从类缓存中取 `hessian` 对应的实现类
2. 如果没有，先触发加载配置

#### 第六步：创建并缓存实例

这里不会每次都 `new`。  
而是通常通过 `SingletonFactory` 获取单例对象，并缓存起来。

这样下次再拿同样扩展名时，就可以直接复用。

### 3. 为什么 `ExtensionLoader` 要缓存

如果每次获取扩展都重新：

1. 读配置文件
2. 反射加载类
3. 创建对象

那效率会很差。

所以它一般会维护两层缓存：

1. 扩展类缓存  
   `name -> class`

2. 扩展实例缓存  
   `name -> instance`

### 4. `ExtensionLoader` 的价值

它是这个项目里最关键的基础设施之一，因为它把“框架支持多种实现”的能力真正落了地。

没有它，框架里的很多地方只能写死：

- 只能 Hessian
- 只能 ZooKeeper
- 只能某一种负载均衡

而有了它，这些能力都能做成插件式切换。

### 背诵版

`ExtensionLoader` 是项目中的扩展加载核心，它的作用是根据扩展接口和扩展名，动态加载并返回对应实现对象。它的工作流程是：先校验接口是否合法并带有 `@SPI`，再读取 `META-INF/extensions` 下的配置文件，解析扩展名和实现类映射并缓存，最后按名称获取实现类，并通过单例工厂创建或复用实例。它的意义在于把序列化、压缩、注册中心、负载均衡等能力做成可插拔扩展。

## 九、`SingletonFactory` 详解

### 1. 它解决什么问题

框架里有很多对象不适合反复创建，比如：

1. 扩展加载器内部要复用的对象
2. 缓存型组件
3. 请求管理器
4. 连接管理器

如果这些对象每次都重新创建，会导致：

1. 资源浪费
2. 状态不一致
3. 缓存失效

所以需要一个统一地方来管理单例。

### 2. 为什么这个项目不完全依赖 Spring

因为这个项目不是所有类都交给 Spring 容器管理。  
有很多基础设施类：

1. 可能在非 Spring 场景就要用
2. 可能在 SPI 加载阶段就要创建
3. 可能本身就是底层框架组件

所以作者自己实现了 `SingletonFactory`，让框架层不强依赖 Spring。

### 3. 它和 `ExtensionLoader` 的关系

`ExtensionLoader` 负责找到“应该用哪个类”，  
`SingletonFactory` 负责决定“这个类的对象怎么创建和复用”。

可以这样记：

1. `ExtensionLoader` 管“找实现”
2. `SingletonFactory` 管“拿实例”

### 4. 为什么单例在这个项目里很重要

因为很多组件具备共享状态，比如：

1. 缓存表
2. 未完成请求表
3. Channel 管理器
4. 扩展加载器

这些都不适合每次都创建新对象。

### 背诵版

`SingletonFactory` 是项目中的统一单例创建工厂，用于保证基础组件在框架运行期间只创建一个实例。这样可以避免资源浪费、缓存失效和状态不一致。它和 `ExtensionLoader` 的关系是：`ExtensionLoader` 负责按名称找到扩展实现，`SingletonFactory` 负责创建和复用对应实例。

## 十、线程池工具知识点

### 1. 为什么线程池工具重要

RPC 框架最终一定会涉及并发处理请求。  
无论是：

1. Socket 版服务端
2. Netty 服务端业务线程池
3. 关闭时的统一释放

线程池都是基础能力。

### 2. 为什么要统一封装线程池工具

如果在项目里到处自己 `Executors.newFixedThreadPool()`，会有几个问题：

1. 参数不统一
2. 命名不统一
3. 难以统一关闭
4. 难以监控和排错

所以项目用：

- `CustomThreadPoolConfig`
- `ThreadPoolFactoryUtil`

统一管理线程池配置和创建逻辑。

### 3. 为什么线程池工具放在 common 里

因为线程池能力不是 RPC 主流程独有，而是全局基础设施。

后续：

1. Socket 模型要用
2. Netty 业务处理可能要用
3. 关闭钩子也会用

所以它放在 `common` 最合理。

### 背诵版

线程池工具属于框架的通用基础能力，而不是某一条主链路独有的逻辑。统一封装线程池配置和创建工具，可以保证参数一致、命名一致、关闭一致，也便于后续 Socket 和 Netty 场景复用。

## 十三、Day 2 必背问答

### 1. 为什么要单独拆出 `rpc-framework-common`？

答：因为这个模块承载的是 RPC 框架的公共基础能力，包括枚举、异常、工具类、SPI 加载器、单例工厂和线程池工具。它们会被核心模块反复复用，不适合和主流程代码混在一起。

### 2. 为什么框架里大量使用枚举？

答：因为枚举可以统一表达固定选项和固定状态，减少硬编码字符串，提升可维护性和扩展性。

### 3. `RpcException` 的作用是什么？

答：`RpcException` 用于统一表达 RPC 主流程中的错误，比如服务未找到、请求响应不匹配和服务调用失败，它把框架层错误和底层 Java 异常区分开了。

### 4. 什么是 SPI？

答：SPI 是一种可插拔扩展机制，框架先定义接口，再通过配置文件把扩展名映射到具体实现类，运行时按名称动态加载对应实现。

### 5. 这个项目里的 SPI 有什么特点？

答：它更接近 Dubbo SPI，支持按名称精确获取实现，并结合缓存和单例工厂来提升性能和保持实例复用。

### 6. `ExtensionLoader` 的核心作用是什么？

答：根据扩展接口和扩展名，动态找到并返回对应实现对象。

### 7. `SingletonFactory` 的作用是什么？

答：统一管理框架中的单例对象，避免重复创建基础设施组件，保证缓存和状态一致。

### 8. 为什么不完全依赖 Spring 单例？

答：因为很多底层组件需要在非 Spring 场景下也能使用，比如 SPI 加载和基础设施对象创建，所以框架实现了自己的单例工厂。

### 9. `ThreadPoolFactoryUtil` 为什么放在 common 里？

答：因为线程池属于框架通用基础能力，不属于某一条具体业务调用链，Socket 和 Netty 场景都可能复用。

### 10. Day 2 对后面几天的意义是什么？

答：Day 2 搭的是整个 RPC 框架的基础设施层，后续进入请求处理、通信、注册发现和动态代理时，都会直接依赖今天复刻的公共能力。

## 十五、Day 2 总结

Day 2 的本质，不是写“很多辅助类”，而是把 RPC 框架真正的基础设施层复刻出来。
